PostScript Basics

  • Adobe's first product (1985-1999)
  • three versions/levels
  • centered on graphics (including text)
  • about 385 built-in operators
  • PS is the precursor to PDF
  • PS and PDF: significant overlap, with differences

Basic Use

  • generate PS as an intermediate output, then convert to PDF
  • code -> PS-> PDF
  • use case: report generation

Convert PS to PDF

  • Ghostscript
  • Adobe's Distiller (part of Acrobat)
  • this operation is trivial (don't worry about it)

Strange Parts

  • postfix notation
  • operand stack (pass args, return data)
  • tokenizes an incoming stream of code
  • no program counter, as in most languages
  • LISPy, with code-as-data
  • dynamic namespace
  • typed, but no type for text or characters

Unpleasant Parts

  • harder to read than average
  • you need to track state in your head
  • minimal tooling
  • very few third-party libraries
  • seemingly out of fashion
  • Eclipse plugin (unsigned code)

Pleasant Parts

  • excellent documentation (Red Book, the Language Reference)
  • fast development
  • scalable vector everything
  • feelings of power, precise control
  • building charts, tables is fairly easy and natural
  • tools convert PS to PDF very quickly

A text file with this content: 
%!PS-Adobe-3.0

/Helvetica 48 selectfont
300 500  moveto
(Hello) show

showpage
Run Ghostscript to read in PS and output PDF: 
 C:\ghostscript\gs10.04.0\bin>
   gswin64c.exe 
   -dNOSAFER 
   -sDEVICE=pdfwrite  
   -o C:\temp\hello.pdf  
   C:\temp\hello.ps

View the output hello.pdf Examples (as above) are almost always hard-coded to specific page dimensions. This a coding horror, because PS is designed for scale-independence. The tasteful way is to define the page format in one place, and to use percentages everywhere else in your code.
